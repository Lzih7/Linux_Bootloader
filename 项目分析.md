**两个工程相互独立**，各自有独立的 Makefile 和链接脚本，但**共享同一份**启动文件和 CMSIS 库。这是嵌入式 Bootloader + App 分区设计的标准做法。

## 2. 硬件内存地址空间

### STM32F401RET6 物理内存布局

```
地址空间（32位）
┌─────────────────────────────────────┐
│  0xFFFFFFFF                         │
│       ...（外设、系统控制等）          │
│  0xE000ED08  ← SCB->VTOR 寄存器     │  ← 中断向量表偏移寄存器
│       ...                           │
│  0x40000000  ← 外设寄存器区          │  ← RCC、GPIO、USART 等
│  0x3FFFFFFF                         │
│       ...                           │
│  0x20017FFF  ← SRAM 结束            │
│  0x20000000  ← SRAM 起始（96KB）    │  ← 运行时变量、堆、栈
│  0x1FFFFFFF                         │
│       ...                           │
│  0x0807FFFF  ← Flash 结束           │
│  0x08010000  ← Application 起始     │  ← App 代码从这里开始
│  0x0800FFFF  ← Bootloader 结束      │
│  0x08000000  ← Flash 起始/Boot起始  │  ← 上电后 CPU 从这里取第一条指令
└─────────────────────────────────────┘
```

### 本项目 Flash 分区方案

```
Flash 总空间：512KB（0x08000000 ~ 0x0807FFFF）

┌──────────────────────────────────────────┐ 0x0807FFFF
│                                          │
│         Application 区域                  │
│         448KB                            │
│         (0x08010000 ~ 0x0807FFFF)        │
│                                          │
├──────────────────────────────────────────┤ 0x08010000  ← APP_START_ADDR
│                                          │
│         Bootloader 区域                   │
│         64KB                             │
│         (0x08000000 ~ 0x0800FFFF)        │
│                                          │
└──────────────────────────────────────────┘ 0x08000000  ← 上电入口
```

### SRAM 分配

```
SRAM：96KB（0x20000000 ~ 0x20017FFF）

Bootloader 运行阶段：
  .data  → 从 Flash 复制过来的初始化变量
  .bss   → 清零的未初始化变量
  Stack  → 从 0x20018000（栈顶）向下增长

Application 运行阶段：
  同上，两者不同时运行，共用全部 SRAM
```

> **关键**：Bootloader 和 Application **不同时运行**，因此 SRAM 可以完全共享，无需分区。

## 4. Makefile 详解

两个工程的 Makefile 结构完全一致，仅在目标名称和链接脚本路径上有差异。以 **Bootloader Makefile** 为例逐行分析。

### 4.1 目标名称与编译开关

```makefile
TARGET = bootloader    # 最终输出文件名前缀：bootloader.elf / .hex / .bin

DEBUG = 1              # 1 = 生成调试信息，0 = 不生成
OPT = -Og              # 优化级别：-Og 是专为调试设计的优化，保留变量/行号信息
                       # 生产环境可改为 -O2 或 -Os（代码尺寸优化）
BUILD_DIR = build      # 编译产物输出目录
```

### 4.2 源文件列表

```makefile
C_SOURCES = \
../bootloader/src/main.c \              # Bootloader 业务逻辑
../common/cmsis/device/system_stm32f4xx.c  # CMSIS 系统初始化

ASM_SOURCES = \
../common/startup/startup_stm32f401xe.s    # 启动汇编文件
```

**注意路径**：Makefile 位于 `bootloader/` 目录下，所以用 `../` 引用上级目录的共享文件。

### 4.3 工具链定义

```makefile
PREFIX = arm-none-eabi-
CC = $(PREFIX)gcc                          # C 编译器
AS = $(PREFIX)gcc -x assembler-with-cpp   # 汇编器（用 gcc 驱动，支持 #include 等预处理）
CP = $(PREFIX)objcopy                      # 格式转换工具
SZ = $(PREFIX)size                         # 大小统计工具

HEX = $(CP) -O ihex      # 转为 Intel HEX 格式
BIN = $(CP) -O binary -S # 转为纯二进制，-S 去除调试符号
```

### 4.4 编译器标志（CFLAGS）

```makefile
# CPU 相关标志（三个标志合并为 MCU）
CPU = -mcpu=cortex-m4          # 指定 ARM Cortex-M4 内核
FPU = -mfpu=fpv4-sp-d16        # 使用硬件 FPU（单精度，16个D寄存器）
FLOAT-ABI = -mfloat-abi=hard   # 硬浮点 ABI：浮点参数通过 FPU 寄存器传递
MCU = $(CPU) -mthumb $(FPU) $(FLOAT-ABI)
#   -mthumb：生成 Thumb-2 指令集（比 ARM 指令集代码更小）
```

```makefile
# 宏定义
C_DEFS = -DSTM32F401xE
# 这个宏在 stm32f4xx.h 中用于条件编译，选择正确的寄存器定义
# 例如：#ifdef STM32F401xE → include "stm32f401xe.h"
```

```makefile
# 头文件搜索路径
C_INCLUDES = \
-I../bootloader/inc \          # 本工程私有头文件（main.h）
-I../common/cmsis/core \       # ARM CMSIS 核心头（core_cm4.h, cmsis_gcc.h 等）
-I../common/cmsis/device       # ST 设备头（stm32f4xx.h, system_stm32f4xx.h）
```

```makefile
# 完整 CFLAGS
CFLAGS = $(MCU) $(C_DEFS) $(C_INCLUDES) $(OPT) -Wall -fdata-sections -ffunction-sections
#   -Wall：开启所有常用警告
#   -fdata-sections：每个变量单独放一个 section（配合链接器 GC 使用）
#   -ffunction-sections：每个函数单独放一个 section（配合链接器 GC 使用）

ifeq ($(DEBUG), 1)
CFLAGS += -g -gdwarf-2
#   -g：生成调试信息
#   -gdwarf-2：使用 DWARF2 格式（GDB 兼容性好）
endif

CFLAGS += -MMD -MP -MF"$(@:%.o=%.d)"
#   -MMD：自动生成 .d 依赖文件（记录 .c 依赖的所有 .h）
#   -MP：为每个头文件生成一个空规则（防止删除头文件后 make 报错）
#   -MF：指定 .d 文件的输出路径
```

### 4.5 链接器标志（LDFLAGS）

```makefile
LDSCRIPT = ../bootloader/ld/STM32F401RET6_BOOTLOADER.ld  # 链接脚本路径

LIBS = -lc -lm -lnosys
#   -lc：链接 newlib C 标准库
#   -lm：链接数学库
#   -lnosys：链接 nosys 存根（提供无 OS 的系统调用空实现，如 _sbrk, _write）

LDFLAGS = $(MCU) -specs=nano.specs -T$(LDSCRIPT) $(LIBDIR) $(LIBS) \
          -Wl,-Map=$(BUILD_DIR)/$(TARGET).map,--cref \
          -Wl,--gc-sections
#   -specs=nano.specs：使用 newlib-nano（体积更小的 C 库，适合嵌入式）
#   -T$(LDSCRIPT)：指定链接脚本
#   -Wl,-Map=...：生成 .map 文件（可以看到每个符号的最终链接地址）
#   -Wl,--cref：在 .map 文件中生成交叉引用表
#   -Wl,--gc-sections：链接时垃圾回收未使用的 section（配合 -ffunction-sections 减小体积）
```

> **`-ffunction-sections` + `--gc-sections` 联动**：编译时每个函数独立成 section，链接时 `--gc-sections` 检查哪些 section 没有被引用，直接丢弃。这是嵌入式工程减小固件体积的标准手段。

### 4.6 构建规则

```makefile
# 对象文件列表生成
OBJECTS = $(addprefix $(BUILD_DIR)/,$(notdir $(C_SOURCES:.c=.o)))
# 例：../bootloader/src/main.c → build/main.o
vpath %.c $(sort $(dir $(C_SOURCES)))
# vpath 告诉 make 去哪些目录找 .c 文件（因为 .o 规则只写文件名，不带路径）

OBJECTS += $(addprefix $(BUILD_DIR)/,$(notdir $(ASM_SOURCES:.s=.o)))
vpath %.s $(sort $(dir $(ASM_SOURCES)))
```

```makefile
# 编译 .c → .o
$(BUILD_DIR)/%.o: %.c Makefile | $(BUILD_DIR)
    $(CC) -c $(CFLAGS) -Wa,-a,-ad,-alms=$(BUILD_DIR)/$(notdir $(<:.c=.lst)) $< -o $@
#   -c：只编译不链接
#   -Wa,-a,-ad,-alms=xxx.lst：同时生成汇编列表文件（.lst），可以看到 C 代码对应的汇编
#   $<：源文件（.c）
#   $@：目标文件（.o）

# 汇编 .s → .o
$(BUILD_DIR)/%.o: %.s Makefile | $(BUILD_DIR)
    $(AS) -c $(CFLAGS) $< -o $@

# 链接所有 .o → .elf
$(BUILD_DIR)/$(TARGET).elf: $(OBJECTS) Makefile
    $(CC) $(OBJECTS) $(LDFLAGS) -o $@
    $(SZ) $@     # 打印各段大小：text/data/bss

# .elf → .hex
$(BUILD_DIR)/%.hex: $(BUILD_DIR)/%.elf | $(BUILD_DIR)
    $(HEX) $< $@

# .elf → .bin
$(BUILD_DIR)/%.bin: $(BUILD_DIR)/%.elf | $(BUILD_DIR)
    $(BIN) $< $@
```

### 4.7 依赖文件自动包含

```makefile
-include $(wildcard $(BUILD_DIR)/*.d)
# 包含所有 .d 文件（头文件依赖）
# 前面的 `-` 表示即使文件不存在也不报错（第一次编译时 .d 还不存在）
```

### 4.8 Application Makefile 与 Bootloader 的差异

| 项目 | Bootloader | Application |
|------|-----------|-------------|
| `TARGET` | `bootloader` | `application` |
| `LDSCRIPT` | `STM32F401RET6_BOOTLOADER.ld` | `STM32F401RET6_APPLICATION.ld` |
| `C_SOURCES` | `bootloader/src/main.c` | `application/src/main.c` |
| `C_INCLUDES` | `-I../bootloader/inc` | `-I../application/inc` |

其余标志完全相同。

## 5. 链接脚本 .ld 详解

链接脚本（Linker Script）是嵌入式编译中最关键的配置文件之一。它告诉链接器：**把哪些代码/数据放到哪块内存的哪个地址**。

### 5.1 Bootloader 链接脚本

**文件**：`bootloader/ld/STM32F401RET6_BOOTLOADER.ld`

#### 入口点与栈顶符号

```ld
ENTRY(Reset_Handler)
/* 告诉链接器：程序入口是 Reset_Handler 函数
   这不影响硬件行为（硬件从向量表读取），但 GDB 调试时会从这里开始 */

_estack = ORIGIN(RAM) + LENGTH(RAM);
/* 定义栈顶地址符号：0x20000000 + 96*1024 = 0x20018000
   这个符号会被启动文件和向量表第0项引用 */

_Min_Heap_Size  = 0x200;   /* 最小堆空间：512 字节 */
_Min_Stack_Size = 0x400;   /* 最小栈空间：1024 字节 */
```

#### MEMORY 区域定义

```ld
MEMORY
{
  RAM   (xrw) : ORIGIN = 0x20000000, LENGTH = 96K
  FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 64K
}
```

**权限标志含义**：
- `r`：可读
- `w`：可写
- `x`：可执行
- `xrw`：RAM 可读可写可执行
- `rx`：FLASH 可读可执行，不可写（ROM 属性）

**Bootloader 的 Flash 范围**：`0x08000000` 起，`64KB` 长，即 `0x08000000 ~ 0x0800FFFF`。

#### SECTIONS 段布局

**`.isr_vector` 段（中断向量表）**

```ld
.isr_vector :
{
  . = ALIGN(4);
  KEEP(*(.isr_vector))   /* 保留中断向量表，即使没有代码引用它也不被 GC 删除 */
  . = ALIGN(4);
} >FLASH
```

- `.` 是"当前地址计数器"（Location Counter）
- `ALIGN(4)` 对齐到 4 字节边界
- `KEEP()` 防止 `--gc-sections` 把向量表当作"未使用"删掉
- `>FLASH` 表示该段放入 FLASH 区域
- 由于 FLASH 从 `0x08000000` 开始，向量表就放在 `0x08000000`，这正是 Cortex-M4 上电后读取的位置

**`.text` 段（程序代码）**

```ld
.text :
{
  . = ALIGN(4);
  *(.text)           /* 所有目标文件的 .text section（普通函数代码） */
  *(.text*)          /* .text.functionName 等子 section（-ffunction-sections 产生的） */
  *(.glue_7)         /* ARM 到 Thumb 的胶合代码（混合指令集时需要） */
  *(.glue_7t)        /* Thumb 到 ARM 的胶合代码 */
  *(.eh_frame)       /* 异常处理帧信息（C++ 异常或 unwind） */
  KEEP (*(.init))    /* 初始化例程（C 运行时启动） */
  KEEP (*(.fini))    /* 析构例程 */
  . = ALIGN(4);
  _etext = .;        /* 导出符号：代码段结束地址（启动文件用来定位 .data 的初始值） */
} >FLASH
```

**`.rodata` 段（只读常量）**

```ld
.rodata :
{
  . = ALIGN(4);
  *(.rodata)    /* 字符串常量、const 变量等 */
  *(.rodata*)
  . = ALIGN(4);
} >FLASH
```

**`.ARM.extab` 和 `.ARM` 段（异常处理）**

```ld
.ARM.extab : { *(.ARM.extab* .gnu.linkonce.armextab.*) } >FLASH
.ARM : {
  __exidx_start = .;
  *(.ARM.exidx*)   /* 异常处理索引表（C++ 异常 / stack unwind 用） */
  __exidx_end = .;
} >FLASH
```

这两个段是 ARM EABI 的标准，即使不用 C++ 异常也需要保留（newlib 的 `_exit` 等可能依赖它）。

**`.preinit_array`、`.init_array`、`.fini_array` 段**

```ld
.init_array :
{
  PROVIDE_HIDDEN (__init_array_start = .);
  KEEP (*(SORT(.init_array.*)))
  KEEP (*(.init_array*))
  PROVIDE_HIDDEN (__init_array_end = .);
} >FLASH
```

这些段存放 C++ 全局构造函数和析构函数的指针数组。启动代码中的 `__libc_init_array()` 会遍历 `__init_array_start ~ __init_array_end`，依次调用所有构造函数。

**`.data` 段（初始化变量）**

```ld
_sidata = LOADADDR(.data);   /* 符号：.data 在 Flash 中的存储地址 */

.data :
{
  . = ALIGN(4);
  _sdata = .;         /* 符号：.data 在 RAM 中的起始地址 */
  *(.data)
  *(.data*)
  *(.RamFunc)         /* 需要在 RAM 中运行的函数（__attribute__((section(".RamFunc")))） */
  *(.RamFunc*)
  . = ALIGN(4);
  _edata = .;         /* 符号：.data 在 RAM 中的结束地址 */
} >RAM AT> FLASH
```

**`>RAM AT> FLASH` 是关键**：
- `>RAM`：运行时地址（VMA，Virtual Memory Address）在 RAM
- `AT> FLASH`：加载地址（LMA，Load Memory Address）在 FLASH

这意味着：编译器生成的 `.data` 初始值**存储在 Flash 里**，但程序运行时变量**在 RAM 中访问**。启动代码负责在上电时将这段数据从 Flash 复制到 RAM。

`_sidata`、`_sdata`、`_edata` 三个符号正是为启动代码服务的：
```
Flash：[ _sidata → _sidata+(edata-sdata) ] 存放初始值
RAM：  [ _sdata  → _edata              ] 运行时访问地址
```

**`.bss` 段（未初始化变量）**

```ld
.bss :
{
  _sbss = .;
  __bss_start__ = _sbss;
  *(.bss)
  *(.bss*)
  *(COMMON)    /* 未初始化的全局变量（老式 C 编译器生成） */
  . = ALIGN(4);
  _ebss = .;
  __bss_end__ = _ebss;
} >RAM
```

`.bss` 只在 RAM 中分配空间，Flash 里不存储任何内容（全是 0）。启动代码负责将 `_sbss ~ _ebss` 这段 RAM 清零。

**`._user_heap_stack` 段（堆栈空间检查）**

```ld
._user_heap_stack :
{
  . = ALIGN(8);
  PROVIDE ( end = . );
  PROVIDE ( _end = . );
  . = . + _Min_Heap_Size;    /* 预留 0x200 字节堆空间 */
  . = . + _Min_Stack_Size;   /* 预留 0x400 字节栈空间 */
  . = ALIGN(8);
} >RAM
```

这个段**不放实际数据**，仅用于**检查 RAM 是否足够**。如果 `.bss` 之后剩余 RAM 不足 `_Min_Heap_Size + _Min_Stack_Size`，链接时会报错。`_end` 符号是 newlib `_sbrk()` 的堆起始地址。

**`/DISCARD/` 丢弃段**

```ld
/DISCARD/ :
{
  libc.a ( * )
  libm.a ( * )
  libgcc.a ( * )
}
```

丢弃标准库中链接进来的调试/异常信息段，减小固件体积。

### 5.2 Application 链接脚本的差异

**文件**：`application/ld/STM32F401RET6_APPLICATION.ld`

只有 MEMORY 块不同：

```ld
MEMORY
{
  RAM   (xrw) : ORIGIN = 0x20000000, LENGTH = 96K    /* 相同 */
  FLASH (rx)  : ORIGIN = 0x08010000, LENGTH = 448K   /* 不同！*/
}
```

- Flash 起始地址从 `0x08000000` 变为 `0x08010000`（偏移 64KB）
- Flash 长度从 `64K` 变为 `448K`（剩余所有 Flash）

**这一个改变影响极大**：所有代码和向量表都会被链接到 `0x08010000` 开始的地址，Bootloader 跳转时也必须跳到这里。

### 5.3 链接脚本中符号与启动代码的对应关系

```
链接脚本符号          启动代码用途
─────────────────────────────────────────────────────
_estack          → 向量表第0项（初始 SP 值）
_sidata          → .data 在 Flash 中的初始���位置
_sdata / _edata  → .data 在 RAM 中的目标范围（复制目标）
_sbss  / _ebss   → .bss  在 RAM 中的范围（清零范围）
_end             → newlib 堆起始地址（_sbrk 使用）
```

## 6. 启动文件 startup_stm32f401xe.s 详解

**文件**：`common/startup/startup_stm32f401xe.s`

启动文件是上电后**第一段被执行的代码**。它在调用 `main()` 之前完成 C 运行环境的初始化。

### 6.1 汇编指令声明

```asm
.syntax unified    /* 使用统一汇编语法（ARM + Thumb 混合语法） */
.cpu cortex-m4     /* 目标 CPU，影响指令集选择 */
.fpu softvfp       /* 使用软件 FPU（注意：Makefile 里用的是 hard FPU，这里是汇编器设置） */
.thumb             /* 生成 Thumb 指令集代码 */
```

### 6.2 链接脚本符号引用

```asm
.word  _sidata   /* 导入链接脚本中定义的符号（Flash 中 .data 初始值的地址） */
.word  _sdata    /* RAM 中 .data 段起始地址 */
.word  _edata    /* RAM 中 .data 段结束地址 */
.word  _sbss     /* RAM 中 .bss 段起始地址 */
.word  _ebss     /* RAM 中 .bss 段结束地址 */
```

### 6.3 Reset_Handler（核心启动流程）

```asm
.section .text.Reset_Handler    /* 放入独立 section，支持 GC */
.weak   Reset_Handler           /* weak 符号，可被用户重定义覆盖 */
.type   Reset_Handler, %function

Reset_Handler:
  ldr sp, =_estack              /* 第1步：设置栈指针 SP = 0x20018000 */

  bl  SystemInit                /* 第2步：调用 SystemInit()
                                   → 使能 FPU
                                   → 重置 RCC 时钟到默认（HSI 16MHz）
                                   → 设置 SCB->VTOR = FLASH_BASE（0x08000000） */

/* 第3步：将 .data 初始值从 Flash 复制到 RAM */
  ldr r0, =_sdata               /* r0 = RAM 目标起始地址 */
  ldr r1, =_edata               /* r1 = RAM 目标结束地址 */
  ldr r2, =_sidata              /* r2 = Flash 源数据起始地址 */
  movs r3, #0                   /* r3 = 字节偏移量，初始为0 */
  b LoopCopyDataInit

CopyDataInit:
  ldr r4, [r2, r3]              /* 从 Flash 读4字节：r4 = *(r2+r3) */
  str r4, [r0, r3]              /* 写入 RAM：*(r0+r3) = r4 */
  adds r3, r3, #4               /* 偏移量 +4（每次复制4字节） */

LoopCopyDataInit:
  adds r4, r0, r3               /* r4 = 当前 RAM 写入地址 */
  cmp r4, r1                    /* 是否到达 _edata？ */
  bcc CopyDataInit              /* 未到达则继续（bcc = branch if carry clear，即 <） */

/* 第4步：将 .bss 清零 */
  ldr r2, =_sbss
  ldr r4, =_ebss
  movs r3, #0
  b LoopFillZerobss

FillZerobss:
  str r3, [r2]                  /* 写0到当前地址 */
  adds r2, r2, #4               /* 地址 +4 */

LoopFillZerobss:
  cmp r2, r4                    /* 是否到达 _ebss？ */
  bcc FillZerobss

  bl __libc_init_array          /* 第5步：调用 C++ 全局构造函数 */
  bl main                       /* 第6步：调用 main() */
  bx lr                         /* main() 返回后（理论上不应到这里） */
```

### 6.4 Default_Handler（默认中断处理）

```asm
.section .text.Default_Handler,"ax",%progbits
Default_Handler:
Infinite_Loop:
  b Infinite_Loop               /* 未处理中断：死循环（便于调试时定位问题） */
```

### 6.5 中断向量表 g_pfnVectors

```asm
.section .isr_vector,"a",%progbits    /* "a" = allocatable，会被链接到 .isr_vector section */
.type g_pfnVectors, %object

g_pfnVectors:
  .word _estack                 /* 偏移 0x00：初始栈指针（SP） */
  .word Reset_Handler           /* 偏移 0x04：复位向量（CPU 上电后跳转地址） */
  .word NMI_Handler             /* 偏移 0x08：不可屏蔽中断 */
  .word HardFault_Handler       /* 偏移 0x0C：硬件错误 */
  .word MemManage_Handler       /* 偏移 0x10：内存管理错误 */
  .word BusFault_Handler        /* 偏移 0x14：总线错误 */
  .word UsageFault_Handler      /* 偏移 0x18：使用错误 */
  .word 0                       /* 偏移 0x1C~0x28：保留 */
  .word 0
  .word 0
  .word 0
  .word SVC_Handler             /* 偏移 0x2C：系统调用 */
  .word DebugMon_Handler        /* 偏移 0x30：调试监视器 */
  .word 0                       /* 保留 */
  .word PendSV_Handler          /* 偏移 0x38：可挂起系统调用（RTOS 上下文切换） */
  .word SysTick_Handler         /* 偏移 0x3C：系统定时器 */
  /* 0x40 开始：外部中断（IRQ0 ~ IRQn） */
  .word WWDG_IRQHandler
  ...（共 98 个中断）
```

**向量表布局规则**：Cortex-M4 上电时，硬件自动从 `SCB->VTOR` 指向的地址读取：
- 第0个字（offset 0x00）→ 加载到 SP（栈指针）
- 第1个字（offset 0x04）→ 加载到 PC（程序计数器），即跳转到 Reset_Handler

### 6.6 弱符号（Weak Symbol）机制

```asm
.weak   NMI_Handler
.thumb_set NMI_Handler, Default_Handler
```

所有中断处理函数都定义为 **weak 符号**，默认指向 `Default_Handler`（死循环）。
如果用户在自己的 .c 文件里定义了同名函数（如 `void SysTick_Handler(void) {...}`），链接时用户的实现会**覆盖** weak 符号，而不报重定义错误。

### 7.3 system_stm32f4xx.c 详解

#### `SystemInit()` 函数

```c
void SystemInit(void)
{
  /* 1. 启用 FPU（Cortex-M4F 的硬件浮点单元） */
  #if (__FPU_PRESENT == 1) && (__FPU_USED == 1)
    SCB->CPACR |= ((3UL << 10*2)|(3UL << 11*2));
    /* CP10 和 CP11 设为 Full Access（允许在任意权限级别使用 FPU） */
  #endif

  /* 2. 重置 RCC 时钟到默认状态（使用内部 HSI 16MHz） */
  RCC->CR |= 0x00000001;          /* 使能 HSI */
  RCC->CFGR = 0x00000000;         /* 系统时钟切换回 HSI */
  RCC->CR &= 0xFEF6FFFF;          /* 关闭 HSE、CSS、PLL */
  RCC->PLLCFGR = 0x24003010;      /* 复位 PLL 配置 */
  RCC->CR &= 0xFFFBFFFF;          /* 关闭 HSE Bypass */
  RCC->CIR = 0x00000000;          /* 清除所有时钟中断标志 */

  /* 3. 设置中断向量表位置 */
  SCB->VTOR = FLASH_BASE | VECT_TAB_OFFSET;
  /* FLASH_BASE = 0x08000000, VECT_TAB_OFFSET = 0x00
     所以 SCB->VTOR = 0x08000000
     这对 Bootloader 是正确的；Application 需要之后再手动改为 0x08010000 */
}
```

**重要**：`SystemInit()` 由启动文件在 `Reset_Handler` 最开始调用（在复制 .data、清零 .bss 之前），因此不能依赖任何全局变量。

#### `SystemCoreClockUpdate()` 函数

```c
void SystemCoreClockUpdate(void)
{
  /* 读取 RCC->CFGR 的 SWS 位，判断当前系统时钟来源 */
  switch (RCC->CFGR & RCC_CFGR_SWS)
  {
    case 0x00: SystemCoreClock = HSI_VALUE; break;  /* HSI：16MHz */
    case 0x04: SystemCoreClock = HSE_VALUE; break;  /* HSE：外部晶振 */
    case 0x08: /* PLL：计算 PLL 输出频率 */
      pllvco = (HSI_VALUE / pllm) * plln;
      SystemCoreClock = pllvco / pllp;
      break;
  }
  /* 再除以 AHB 预分频系数（HPRE） */
  SystemCoreClock >>= AHBPrescTable[(RCC->CFGR & RCC_CFGR_HPRE) >> 4];
}
```

`SystemCoreClock` 是一个全局变量，存储当前 CPU 主频，供 delay、串口波特率计算等使用。

## 8. Bootloader 源码详解

**文件**：`bootloader/src/main.c`

### 8.1 main() 流程

```c
int main(void)
{
  system_init();           // 初始化时钟（HSI 16MHz）
  gpio_init();             // 初始化 PA5 LED

  if (validate_application() == BOOTLOADER_OK) {
    // App 有效：LED 快闪 1 秒（100ms 间隔 × 10次），然后跳转
    boot_timeout = BOOT_TIMEOUT_MS;  // 1000ms
    while (boot_timeout > 0) {
      led_toggle();
      delay_ms(100);
      boot_timeout -= 100;
    }
    bootloader_jump_to_app();
  } else {
    // App 无效：LED 常亮，死循环
    bootloader_error_loop();
  }
}
```

### 8.2 validate_application() — 应用程序合法性校验

```c
static int32_t validate_application(void)
{
  // 读取 App 向量表的前两个字
  uint32_t app_stack_ptr    = *(volatile uint32_t *)(APP_START_ADDR);      // 0x08010000
  uint32_t app_reset_handler = *(volatile uint32_t *)(APP_START_ADDR + 4); // 0x08010004

  // 校验1：栈指针必须在 SRAM 范围内
  if (app_stack_ptr < SRAM_BASE || app_stack_ptr > (SRAM_END + 1))
    return BOOTLOADER_ERR_INVALID_APP;
  // SRAM_BASE = 0x20000000, SRAM_END = 0x20017FFF

  // 校验2：复位处理程序地址必须在 App Flash 范围内
  if (app_reset_handler < APP_START_ADDR || app_reset_handler > APP_END_ADDR)
    return BOOTLOADER_ERR_INVALID_APP;
  // APP_START_ADDR = 0x08010000, APP_END_ADDR = 0x0807FFFF

  // 校验3：排除全0xFF（未烧录）和全0x00（擦除异常）
  if (app_stack_ptr == 0xFFFFFFFF || app_stack_ptr == 0x00000000)
    return BOOTLOADER_ERR_NO_APP;

  return BOOTLOADER_OK;
}
```

**校验逻辑图解**：

```
Flash[0x08010000] = App 向量表第0项（初始 SP）
  → 必须在 0x20000000 ~ 0x20018000 之间（SRAM 地址）

Flash[0x08010004] = App 向量表第1项（Reset_Handler 地址）
  → 必须在 0x08010000 ~ 0x0807FFFF 之间（App Flash 地址）
  → 且 Thumb 指令地址最低位为 1（实际地址 = 值 & ~1）
```

### 8.3 bootloader_jump_to_app() — 跳转到应用程序

```c
static void bootloader_jump_to_app(void)
{
  typedef void (*pFunction)(void);   // 定义无参无返回值函数指针类型

  uint32_t app_stack_ptr    = *(volatile uint32_t *)(APP_START_ADDR);
  uint32_t app_reset_handler = *(volatile uint32_t *)(APP_START_ADDR + 4);
  pFunction app_entry;

  __disable_irq();               // 1. 关闭全局中断（防止跳转过程被打断）

  __set_MSP(app_stack_ptr);      // 2. 将 MSP（主栈指针）设为 App 的栈顶地址
                                 //    App 的栈指针来自 App 链接脚本的 _estack

  SCB->VTOR = APP_START_ADDR;    // 3. 重定位中断向量表到 App 区域（0x08010000）
                                 //    之后发生的所有中断都会查 App 的向量表

  __enable_irq();                // 4. 重新使能中断

  app_entry = (pFunction)(app_reset_handler);
  app_entry();                   // 5. 跳转！调用 App 的 Reset_Handler
                                 //    从此 Bootloader 代码不再运行
}
```

**跳转时序详解**：

```
Bootloader 运行中
    │
    ├─ __disable_irq()         ← 防止切换期间触发中断
    ├─ __set_MSP(0x20018000)   ← 切换栈到 App 的栈（本例相同，但需要正式设置）
    ├─ SCB->VTOR = 0x08010000  ← 中断向量表切换到 App
    ├─ __enable_irq()          ← 开中断
    └─ app_entry()             ← 执行 App 的 Reset_Handler
           │
           ├─ sp = _estack     ← App 重新设置栈（startup.s 第一行）
           ├─ SystemInit()     ← App 的 SystemInit（VTOR 再次被设为 0x08000000！）
           │                      ← ⚠ 注意：这是个坑，见 Application 源码章节
           ├─ 复制 .data
           ├─ 清零 .bss
           ├─ __libc_init_array
           └─ main()           ← App 的 main 开始运行
```

### 8.4 GPIO 寄存器直接操作详解

```c
void gpio_init(void)
{
  RCC->AHB1ENR |= RCC_AHB1ENR_GPIOAEN;   // 使能 GPIOA 时钟（位0）

  // MODER 寄存器：每引脚2位，PA5 → bits[11:10]
  GPIOA->MODER &= ~(3 << (5 * 2));  // 清除 PA5 模式：11→00
  GPIOA->MODER |=  (1 << (5 * 2));  // 设置 PA5 为输出：00→01

  GPIOA->OTYPER &= ~(1 << 5);       // PA5 推挽输出（0=推挽，1=开漏）

  // OSPEEDR：每引脚2位，10=高速
  GPIOA->OSPEEDR &= ~(3 << (5 * 2));
  GPIOA->OSPEEDR |=  (2 << (5 * 2));  // 10 = High speed

  GPIOA->PUPDR &= ~(3 << (5 * 2));  // 无上下拉（00）

  GPIOA->BSRR = (1 << (5 + 16));    // 复位 PA5（LED 初始关闭）
  // BSRR：bits[15:0] 置位，bits[31:16] 复位
  // (1 << 5)    → 置位 PA5（LED 亮）
  // (1 << 5+16) → 复位 PA5（LED 灭）
}
```

## 9. Application 源码详解

**文件**：`application/src/main.c`

### 9.1 main() — 第一件事：重定位向量表

```c
void main(void)
{
  /* ⚠ 必须在最开始重定位向量表 */
  __disable_irq();
  SCB->VTOR = 0x08010000;   // 将向量表指向 App 自己的向量表
  __enable_irq();

  // 之后才能使用中断（SysTick 等）
  system_init();
  gpio_init();

  while (1) {
    led_toggle();
    delay_ms(LED_BLINK_DELAY_MS);  // 500ms，比 Bootloader 慢 5 倍
  }
}
```

**为什么 App 要再次设置 VTOR？**

因为 `system_init()` 里调用了 `SystemCoreClockUpdate()`，而 `system_stm32f4xx.c` 的 `SystemInit()` 是在 `Reset_Handler` 中被调用的，`SystemInit()` 会将 `SCB->VTOR` 设回 `FLASH_BASE`（0x08000000）。

实际上 Bootloader 跳转时已经设过 VTOR，但由于 App 的 Reset_Handler 里会调用 SystemInit() 再次覆盖，所以 App 的 main() 开头要**显式重新设置** VTOR。

### 9.2 UART2 初始化（可选调试接口）

```c
void uart_init(void)
{
  // PA2(TX) / PA3(RX) 配置为 AF7（USART2）
  GPIOA->AFR[0] |= (7 << (2 * 4));   // PA2 → AF7
  GPIOA->AFR[0] |= (7 << (3 * 4));   // PA3 → AF7

  GPIOA->MODER |= (2 << (2 * 2));    // PA2 → 复用功能模式（10）
  GPIOA->MODER |= (2 << (3 * 2));    // PA3 → 复用功能模式（10）

  USART2->CR1 &= ~USART_CR1_UE;      // 先禁用 USART

  // BRR = 总线时钟 / 波特率 = 16MHz / 115200 ≈ 139
  USART2->BRR = 139;

  USART2->CR1 |= USART_CR1_TE;       // 使能发送
  USART2->CR1 |= USART_CR1_RE;       // 使能接收
  USART2->CR1 |= USART_CR1_UE;       // 使能 USART
}
```

  # ELF 中包含地址信息：0x08010000（由链接脚本决定）
  -c "verify_image application/build/application.elf" \
  -c "reset run" \
  -c "shutdown"
```

**ELF 格式的优势**：ELF 文件包含每个段的目标加载地址（LMA），OpenOCD 会自动将各段写入正确的 Flash 地址，无需手动指定地址。

**内部执行步骤**（按依赖顺序）：

```
步骤1：创建 build/ 目录

  arm-none-eabi-gcc -c \

### 12.3 编译产物内存占用

运行 `arm-none-eabi-size build/bootloader.elf` 输出：

```
   text    data     bss     dec     hex filename
   XXXX      XX     XXX    XXXX    XXXX build/bootloader.elf
```

- `text`：代码段 + 只读数据（占 Flash）
- `data`：初始化变量（占 Flash 存储 + RAM 运行）
- `bss`：未初始化变量（只占 RAM，不占 Flash）
- Flash 占用 = text + data
- RAM 占用 = data + bss + stack + heap

## 13. 完整启动流程（上电到 main 运行）

### 13.1 Bootloader 启动序列

```
上电 / 复位
    │
    ▼
[硬件自动]
SCB->VTOR = 0x08000000（复位默认值）
SP  ← Flash[0x08000000] = _estack = 0x20018000
PC  ← Flash[0x08000004] = Reset_Handler 地址
    │
    ▼
[startup_stm32f401xe.s: Reset_Handler]
1. ldr sp, =_estack        → SP = 0x20018000（再次确认栈顶）
2. bl SystemInit           → 使能 FPU，重置 RCC，SCB->VTOR = 0x08000000
3. 复制 .data 到 RAM       → 初始化全局变量
4. 清零 .bss               → 清零未初始化变量
5. bl __libc_init_array    → 调用 C++ 全局构造函数
6. bl main                 → 进入 Bootloader main()
    │
    ▼
[bootloader/src/main.c: main()]
7. system_init()           → 确认 HSI 时钟，调用 SystemCoreClockUpdate()
8. gpio_init()             → 配置 PA5 为输出
9. validate_application()  → 读取 0x08010000 处的向量表，检查有效性
    │
    ├─── [无效] ───▶ bootloader_error_loop() → LED 常亮，死循环
    │
    └─── [有效]
         10. LED 快闪 1 秒（100ms × 10次）
         11. bootloader_jump_to_app()
              a. __disable_irq()
              b. __set_MSP(App栈顶)
              c. SCB->VTOR = 0x08010000
              d. __enable_irq()
              e. 跳转到 App Reset_Handler
```

### 13.2 Application 启动序列

```
[从 Bootloader 跳转而来，当前 SCB->VTOR = 0x08010000]
    │
    ▼
[startup_stm32f401xe.s: Reset_Handler（同一份文件，但链接到 App 地址）]
1. ldr sp, =_estack        → SP = 0x20018000（App 链接脚本定义的栈顶）
2. bl SystemInit           → ⚠ 再次将 SCB->VTOR 设为 0x08000000！
3. 复制 .data 到 RAM       → 初始化 App 的全局变量
4. 清零 .bss               → 清零 App 的未初始化变量
5. bl __libc_init_array
6. bl main                 → 进入 Application main()
    │
    ▼
[application/src/main.c: main()]
7. __disable_irq()
   SCB->VTOR = 0x08010000  → ⚠ 修复 SystemInit 对 VTOR 的覆盖！
   __enable_irq()
8. system_init()           → 确认时钟
9. gpio_init()             → 配置 PA5
10. while(1): LED 慢闪（500ms）
```

## 14. 关键机制深度解析

### 14.1 中断向量表重定位（VTOR）

`SCB->VTOR`（位于地址 `0xE000ED08`）是 Cortex-M4 的**中断向量表偏移寄存器**。

```
VTOR 约束：
- 必须是 2^N 字节对齐（N ≥ 7，即至少 128 字节对齐）
- 本项目：0x08000000（512MB 对齐，远超要求）��� 0x08010000（64KB 对齐）
- 中断发生时，CPU 从 SCB->VTOR + (中断号 × 4) 处读取处理函数地址
```

**如果 App 不设置 VTOR 会怎样？**

若 VTOR 仍指向 `0x08000000`（Bootloader），当 App 触发任何中断（包括 SysTick），CPU 会查 Bootloader 的向量表，跳转到 Bootloader 的中断处理函数（或 Default_Handler 死循环），导致程序崩溃。

### 14.2 LMA vs VMA（加载地址 vs 运行地址）

```
Flash（存储）                    RAM（运行）
┌────────────────────┐           ┌──────────────┐
│ .text (代码)       │           │              │
│ VMA=LMA=0x08000000 │           │ .data (变量)  │
├────────────────────┤           │ VMA=0x20000000│
│ .rodata (常量)     │   复制 ──►│ LMA=Flash中  │
│ VMA=LMA=0x0800xxxx │           ├──────────────┤
├────────────────────┤           │ .bss (清零)   │
│ .data 初始值       │           │ VMA=0x2000xxxx│
│ LMA=0x0800yyyy     │           └──────────────┘
└────────────────────┘
```

链接脚本中 `.data >RAM AT> FLASH` 这一语法正是描述这种"两地址"情况：
- **LMA**（Load Memory Address）= FLASH：固化在 Flash 里
- **VMA**（Virtual Memory Address）= RAM：运行时在 RAM 里访问

### 14.3 函数指针跳转机制

```c
// App 向量表第1项（Reset_Handler 地址）存储为 Thumb 地址
// Thumb 地址最低位为 1（表示 Thumb 指令集）
// 例如：0x08010001（实际代码地址是 0x08010000）

uint32_t app_reset_handler = *(volatile uint32_t *)(0x08010004);
// app_reset_handler = 0x08010001（Thumb 标志位）

pFunction app_entry = (pFunction)(app_reset_handler);
app_entry();
// CPU 识别最低位为1，以 Thumb 模式执行 0x08010000 处的代码
```

### 14.4 --gc-sections 垃圾回收原理

```
编译时（-ffunction-sections -fdata-sections）：
每个函数 → 独立 section（如 .text.uart_init）
每个变量 → 独立 section（如 .data.tick_counter）

链接时（--gc-sections）：
从入口点（Reset_Handler）开始追踪引用关系：
  Reset_Handler → SystemInit → main → gpio_init → ...

未被引用的 section（如未使用的库函数）→ 直接丢弃

效果：最终固件只包含实际使用的代码和数据
```

### 14.5 nano.specs 与标准 newlib 的区别

| 特性 | newlib（标准） | newlib-nano（-specs=nano.specs） |
|------|--------------|----------------------------------|
| printf 浮点 | 支持 | 不支持（需要 -u _printf_float） |
| 代码大小 | 较大 | 较小（约减少 30%） |
| 适用场景 | 资源充足 | 嵌入式资源受限 |
| malloc | 完整实现 | 简化实现 |
